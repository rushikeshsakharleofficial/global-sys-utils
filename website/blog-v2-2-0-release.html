v2.2.0 — Adaptive Upload Throttle & conf.d Job System

June 2026

What's new

v2.2.0 is the largest feature release since the initial daemon mode. It ships two major capabilities: an adaptive bandwidth throttle for cloud uploads, and a modular conf.d job system that lets each service team own its rotation policy independently.

Adaptive Upload Throttle

Previous versions uploaded archived log files to S3 or GCS using a fixed concurrency limit. On busy servers this caused OOM events during peak backup windows — too many uploads competing with the application workload for network bandwidth and CPU.

The new throttle samples live CPU usage and available RAM every 5 seconds. If either metric is above a configurable threshold, the upload worker count is reduced. When resources recover, concurrency scales back up automatically.

# Configure in global.conf UPLOAD_MAX_WORKERS=8 UPLOAD_CPU_THRESHOLD=80 UPLOAD_RAM_THRESHOLD_MB=512 # Workers auto-reduce when CPU > 80% or free RAM < 512MB

The throttle is enabled by default. Set UPLOAD_MAX_WORKERS=1 to disable adaptive scaling and pin concurrency.

conf.d Job System

In v2.1.x, a single global config file defined all rotation and cloud parameters. For deployments with 10+ services this became a coordination bottleneck — every team needed to edit the same file.

v2.2.0 introduces /etc/global-sys-utils/global.conf.d/. Each .conf file in this directory is an independent rotation + cloud job. The daemon discovers and runs all jobs concurrently on its schedule.

# /etc/global-sys-utils/global.conf.d/api-service.conf LOG_DIR=/var/log/api CLOUD_PROVIDER=aws S3_BUCKET=api-log-archives RETENTION_DAYS=90 # /etc/global-sys-utils/global.conf.d/worker-service.conf LOG_DIR=/var/log/worker CLOUD_PROVIDER=gcp GCS_BUCKET=worker-log-archives RETENTION_DAYS=30

Each job runs isolated — a failure in one conf file does not affect other jobs. Add a new service by dropping a .conf file; remove it by deleting the file. No daemon restart required (the daemon polls conf.d on each cycle).

Package updates

DEB and RPM packages are available for amd64 and arm64. Post-install scripts automatically install Python dependencies and configure systemd units.

# Debian / Ubuntu amd64 wget https://github.com/rushikeshsakharleofficial/global-sys-utils/releases/download/v2.2.0/global-logrotate_2.2.0-1_amd64.deb sudo dpkg -i global-logrotate_2.2.0-1_amd64.deb # RHEL / Fedora x86_64 wget https://github.com/rushikeshsakharleofficial/global-sys-utils/releases/download/v2.2.0/global-logrotate-2.2.0-1.x86_64.rpm sudo rpm -ivh global-logrotate-2.2.0-1.x86_64.rpm

Upgrading from v2.1.x

Your existing /etc/global-sys-utils/global.conf continues to work. The conf.d system is additive — migrate individual services to conf.d files at your own pace. No breaking changes to CLI flags or environment variables.
