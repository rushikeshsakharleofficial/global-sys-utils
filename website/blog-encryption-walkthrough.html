AES-256-GCM Encryption Walkthrough

March 2026

Why encrypt log archives?

Log files often contain PII — IP addresses, user IDs, email addresses, session tokens, and request payloads. Uploading unencrypted archives to S3 or GCS means a bucket misconfiguration or credential leak exposes real user data.

global-sys-utils encrypts each archive with AES-256-GCM before it ever leaves the local filesystem. The cloud provider only ever receives ciphertext. Even with full bucket access, an attacker cannot read the log content without the encryption key.

How it works

  • Each archive gets a random 96-bit nonce — no two archives share an IV even if the content is identical.
  • AES-256-GCM provides both confidentiality and integrity — a tampered archive fails to decrypt.
  • The password is stored only as a SHA-256 hash in /etc/global-sys-utils/. The plaintext is never written to disk after initial setup.
  • At runtime the password is loaded from ~/.global-sys-utils/credentials.ini (mode 0600), or from the LOGROTATE_PASSWORD environment variable.

Step-by-step setup

1. Generate and store a password

# Run once — prompts for a password and stores its hash sudo global-logrotate --pass-gen # Enter password: •••••••••••• # Confirm password: •••••••••••• # ✓ Password hash stored in /etc/global-sys-utils/passwd.sha256 # ✓ Credentials written to /root/.global-sys-utils/credentials.ini (mode 0600)

2. Enable encryption in config

# /etc/global-sys-utils/global.conf ENCRYPT=true LOG_DIR=/var/log/app

3. Rotate with encryption

global-logrotate --encrypt # → /var/log/app/api.log.2026-03-15.gz.enc # → /var/log/app/worker.log.2026-03-15.gz.enc

4. CI / automated environments

# Pass password via environment variable (no credentials.ini needed) LOGROTATE_PASSWORD=your-secret global-logrotate --encrypt # Or export it in your shell profile export LOGROTATE_PASSWORD="$(cat /run/secrets/logrotate-pass)"

5. Decrypt an archive manually

global-logrotate --decrypt api.log.2026-03-15.gz.enc # Enter password: •••••••••••• # → api.log.2026-03-15.gz # Pipe directly to gunzip global-logrotate --decrypt api.log.2026-03-15.gz.enc | gunzip | less

Security considerations

  • Use a unique password per environment (dev / staging / prod). A prod key compromise should not expose staging archives.
  • Rotate the encryption key annually or after any credential leak. Re-encrypt existing archives with the new key before discarding the old one.
  • Set credentials.ini to mode 0600 and owned by the service account running global-logrotate. The post-install script does this automatically.
  • Store the password in a secrets manager (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault) and inject it via LOGROTATE_PASSWORD at runtime rather than storing it on disk at all.
