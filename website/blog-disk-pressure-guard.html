Disk Pressure Guard Deep Dive

April 2026

The problem: log files filling disks

High-throughput services can generate log files faster than scheduled rotation removes them. A single noisy endpoint or a runaway debug flag can fill a disk in hours. When /var/log fills up, applications start throwing write errors, systemd journals stop recording, and SSH may become unavailable if the root filesystem is full.

Standard logrotate runs on a fixed cron schedule — it has no mechanism to respond to real-time disk pressure. By the time the nightly rotation runs, the disk is already full.

Two-level protection

global-sys-utils ships two independent disk guards that work together:

Level 1 — Emergency rotation trigger

The daemon continuously polls disk usage at DISK_CHECK_INTERVAL seconds (default: 60). When used space on the log partition exceeds DISK_CRITICAL_PERCENT, an emergency rotation runs immediately — regardless of the normal schedule.

# /etc/global-sys-utils/global.conf DISK_CRITICAL_PERCENT=90 DISK_CHECK_INTERVAL=60 # Example daemon log output when guard fires: INFO disk usage 91.3% on /var/log — threshold 90% INFO triggering emergency rotation (8 files) INFO freed 4.2 GB in 3.1s — disk usage now 74.1%

Level 2 — Archive write guard

Even when disk usage is below the emergency threshold, creating a compressed archive of a large log file temporarily needs extra space. The archive write guard checks that at least DISK_MIN_FREE_MB megabytes will remain free after the archive is written. If not, it skips archiving that file and logs a warning.

DISK_MIN_FREE_MB=200 # Guard in action: WARN skipping archive of access.log (2.1 GB): only 180 MB free INFO truncating access.log in-place instead

In-place truncation preserves the file descriptor so running processes continue writing without errors. The file is rotated on the next cycle once space is available.

Setup

# Minimal config — add to global.conf DISK_CRITICAL_PERCENT=85 DISK_MIN_FREE_MB=500 DISK_CHECK_INTERVAL=30 # Start the daemon sudo systemctl enable --now global-sys-utils.service # Verify guard is active sudo journalctl -u global-sys-utils.service -f

Tuning recommendations

  • Set DISK_CRITICAL_PERCENT to 10–15% below your actual alert threshold so rotation fires before your monitoring wakes up.
  • Set DISK_MIN_FREE_MB to at least 2× the size of your largest single log file.
  • On NVMe SSDs with fast compression, DISK_CHECK_INTERVAL=15 adds negligible overhead.
  • On slow rotational disks, keep the interval at 60 seconds or higher to avoid polling overhead during heavy I/O.
